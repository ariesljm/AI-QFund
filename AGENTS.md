# AGENTS.md

## Workflow & communication rules
- 依赖与包管理一律使用 `uv`（禁止用 pip/conda 直接装包）。
- **未经明确指示绝不执行 `git commit` / `git push`**（含 amend、强制推送）。
- 全程使用**简体中文**对话。
- 代码注释一律使用**简体中文**；禁止使用全局符号/全局变量（状态通过参数与返回值传递，避免全局可变状态）。

## 行为准则（减少常见 LLM 编码失误）
> 这些准则偏向谨慎而非速度。对于琐碎任务，请自行判断。

### 1. 编码前先思考
**不要想当然，不要掩盖困惑，主动暴露权衡。**
- 明确陈述你的假设；若不确定，先发问。
- 若存在多种理解方式，列出它们，不要默默替用户做选择。
- 若有更简单的方案，说出来；必要时提出反对。
- 若某处不清晰，停下来，指明困惑所在，再询问。

### 2. 简洁优先
**只写能解决问题的最少代码，不做任何投机性扩展。**
- 不写超出需求的功能。
- 不为一次性代码额外抽象。
- 不为未要求的"灵活性"或"可配置性"提前设计。
- 不为不可能发生的场景添加错误处理。
- 若写了 200 行却可压到 50 行，就重写。

自检："资深工程师会觉得这过于复杂吗？"若是，则简化。

### 3. 精准修改
**只动必须改的，只清理自己造成的混乱。**
- 不要顺手"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 沿用既有风格，即便你更倾向另一种写法。
- 发现无关死代码，提一句即可，不要删。
- 你的改动若制造了孤立代码（无用 import/变量/函数），自己清理掉。

检验：每一处改动都应直接对应到用户的请求。

### 4. 目标驱动执行
**先定义成功标准，循环验证直到达成。**
- "加校验" → "先写非法输入的测试，再让它通过"
- "修 bug" → "先写复现测试，再让它通过"
- "重构 X" → "确保重构前后测试均通过"

多步任务先给出简短计划，每步标注验证方式：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```
